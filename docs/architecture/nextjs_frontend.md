# Next.js Frontend Architecture

The visualizer is a modern web application built using **Next.js** (App Router) and **React**.

## The Goal of the UI

The primary purpose of the frontend is to make the staggered data flow of the pipeline visible. While the Rust core computes the math perfectly, looking at text output in a CLI makes it difficult to understand *how* the data moves.

The UI handles:
1.  **Configuration:** Letting the user set vector lengths, weights, and input values.
2.  **Staggering Logic:** Preparing the input data so that it enters the pipeline at the correct cycle.
3.  **Visualization:** Drawing the hardware blocks (PEs) and updating their internal register values on every tick.

## Key Components (Planned)

*Note: The frontend is currently in the planning phase. Below is the intended architecture.*

### 1. WASM Integration
We will use a custom React Hook (e.g., `usePipeline`) to asynchronously load the WASM module generated by Rust. This hook will hold the instance of `DotProductSim` in React state or a ref.

### 2. State Management
React will manage the "Clock Cycle" state. Every time the user clicks "Step" (or if Auto-Play is enabled):
1.  React calculates the correct staggered inputs for the *current* cycle.
2.  React calls the WASM `tick_and_get_state(inputs)` function.
3.  React receives the flattened array of new PE states.
4.  React parses the flat array back into an array of objects and triggers a re-render of the visual components.

### 3. Visual Components
We plan to use **Tailwind CSS** and **Shadcn UI** for styling.

*   **`PEBlock`:** A visual representation of a single Processing Element, showing inputs flowing in from the top/left, the internal weight, and the latched outputs at the bottom/right.
*   **`DataQueue`:** A visual representation of the staggered input vectors waiting on the left side of the pipeline, ready to slide in as the clock ticks.
