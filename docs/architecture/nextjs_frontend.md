# Next.js Frontend Architecture

The visualizer is a modern web application built using **Next.js** (App Router) and **React**.

## The Goal of the UI

The primary purpose of the frontend is to make the staggered data flow of the pipeline visible. While the Rust core computes the math perfectly, looking at text output in a CLI makes it difficult to understand *how* the data moves.

The UI handles:
1.  **Configuration:** Letting the user set vector lengths, weights, and input values.
2.  **Staggering Logic:** Preparing the input data so that it enters the pipeline at the correct cycle.
3.  **Visualization:** Drawing the hardware blocks (PEs) and updating their internal register values on every tick.

## Key Components

### 1. WASM Integration
We use a custom React Hook (`usePipeline`) to asynchronously load the WASM module generated by Rust. This hook holds the instance of `DotProductSim` in React state and manages the lifecycle of the simulation, including automatic resetting when configuration dimensions ($n$, $m$) change.

### 2. State Management
React manages the "Clock Cycle" state and the calculation of staggered inputs. Every time the clock ticks (manually via "Step" or automatically via "Auto-Play"):
1.  React calculates the correct staggered inputs for the *current* cycle.
2.  React calls the WASM `tick(inputs)` function.
3.  React receives the updated PE states directly from the WASM memory buffer.
4.  React parses the flat array back into an array of objects (`PEState`) and triggers a re-render of the visual components, updating the Results Log if a vector calculation has completed.

### 3. Visual Components & Interactivity
The application is styled using **Tailwind CSS** (with the Inter font for a sleek look) and **Shadcn UI** for core components (Cards, Buttons, Inputs, Tables).

*   **`Simulator`**: The main orchestrator component. It provides borderless, shadowed configuration cards allowing users to set vector lengths, number of vectors, weights, and input matrices. It includes **RNG** (Random Number Generation) buttons to quickly populate these fields with random values between -10 and 10.
*   **`PipelineVisualizer`**: The core visualization stage. It renders the vertical stack of Processing Elements.
    *   It features an explicit **Staggered Input Queue** where incoming data vectors are visually aligned diagonally, using 0-padded gray cells to illustrate the exact delay required for the data to enter the systolic array correctly.
    *   Incoming vectors are color-coded, making it easy to track a single vector's progression down the PE pipeline cycle by cycle.
